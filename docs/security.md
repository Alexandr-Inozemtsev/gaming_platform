# Безопасность MVP: токены, античит и защита данных

## Назначение
Этот документ описывает базовую модель угроз и принятые решения по безопасности для MVP.

## Модель угроз простыми словами
1. **Подбор паролей и brute-force логина**
   - Риск: злоумышленник часто вызывает login.
   - Защита: rate-limit на login (`RATE_LIMIT_LOGIN=10` в минуту по IP).

2. **Повторная отправка одного и того же хода (replay)**
   - Риск: клиент повторяет `moveId` и пытается дублировать действие.
   - Защита: idempotency по `moveId` на матч, повтор возвращает `409 Conflict`.

3. **Ход не в свою очередь / клиентский чит на очки**
   - Риск: клиент отправляет ход вне очереди или поддельные очки.
   - Защита: сервер-авторитет — очередь проверяется на сервере, очки рассчитываются только сервером через server RNG.

4. **Сетевые риски в проде**
   - Риск: передача данных без TLS.
   - Защита: политика `REQUIRE_TLS_IN_PROD=true`.

5. **Компрометация паролей**
   - Риск: утечка базы пользователей.
   - Защита: пароль хранится только в виде хэша.

## Примечание по PASSWORD_HASH
Требование: `argon2`, fallback `bcrypt`.

В текущем MVP-коде используется `scrypt` из Node.js `crypto`, потому что в базовом окружении не гарантирована доступность `argon2/bcrypt` без дополнительных зависимостей и сборки.
Это **временное решение** для локального MVP.

### Рекомендованный ручной шаг владельца
- Перевести хэширование на `argon2` (или `bcrypt` fallback),
- выполнить миграцию хэшей пользователей,
- зафиксировать выбор в CI/pipeline окружении.

## Логи безопасности
Сохраняются события:
- подозрительные попытки входа (`SUSPICIOUS_LOGIN`),
- попытки хода не в свой ход (`MOVE_NOT_YOUR_TURN`).

## Что покрыто тестами
- повторный `moveId` -> `409`;
- ход не в свой ход -> `403`;
- rate-limit на login и move -> `429`.
